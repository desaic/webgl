============================================================
IMPLEMENTATION PLAN — Frontend (logic-graph-viewer)
============================================================
Tech: TypeScript, Vite, Three.js (Line2, LineMaterial)
Goal: Upload proof graph JSON → 2.5D interactive viz

The backend is now a multi-domain reasoning engine that
produces a knowledge graph with nodes of many kinds
(FACT, COMPOUND, PORTAL, E-CLASS, RESOURCE, TOOL_RESULT)
and edges of many kinds (RULE, HYPER, INSTANTIATE,
REWRITE, TOOL, DOMAIN, AUDIT). The frontend visualizes
all of this, with domain-aware coloring and resource
previews.

PHASE 1 — Drawing Primitives (modular draw functions)
------------------------------------------------------------
  Each function returns a Three.js Object3D (or Group) for
  a composable scene graph. All positions in world space.

  [ ] 1.1  drawCircle(pos, radius, color, label?)
           → SphereGeometry + optional text sprite
           Accepts optional {segments, materialType} props.

  [ ] 1.2  drawBox(pos, size, color, label?)
           → BoxGeometry + optional text sprite

  [ ] 1.3  drawText(pos, text, fontSize, color)
           → CanvasTexture pre-rendered to sprite/quad.
           Strategy:
           a) TextTextureFactory: offscreen canvas rendering
           b) LRU cache: Map<text_key, THREE.Sprite>
           c) Key = "text|size|color|font"
           d) Return THREE.Sprite with transparent texture
           e) Support multi-line text (split on \n)

  [ ] 1.4  drawLine(from, to, color, width, style?)
           → Line2 + LineGeometry + LineMaterial
           (import from three/addons/lines/)
           style: "solid", "dashed", "dotted"
           → LineMaterial.dashed = true for dashed
           → LineMaterial.dashSize / gapSize for control

  [ ] 1.5  drawArrow(from, to, color, width, headSize?)
           → Line2 segment + small ConeGeometry at tip
           Tip auto-oriented using lookAt(from.sub(to))

  [ ] 1.6  drawPolyline(points[], color, width, closed?)
           → Single Line2 with all points in LineGeometry

  [ ] 1.7  drawCurvedArrow(from, to, color, width, sag?)
           → QuadraticBezierCurve3, tessellate to points,
           feed to Line2. Sag controls arc height.

  [ ] 1.8  drawGrid(size, spacing, color, plane?)
           → GridHelper (default XY plane, configurable)

  [ ] 1.9  drawHighlight(pos1, pos2, color, pulse?)
           → Semi-transparent cylinder between points
           Optionally animate opacity for "pulse" effect.

  [ ] 1.10 drawProofBar(start, end, color, thickness?)
           → BoxGeometry vertical bar for Fitch subproof scope
           (long thin box from y_start to y_end at x_offset)

  [ ] 1.11 drawTableauBranch(point, children[], color)
           → Vertical line from parent, horizontal connector,
           then vertical down to each child. Composition of
           drawLine calls.

  [ ] 1.12 drawCircleNode(pos, radius, color, borderColor?)
           → Flat cylinder (disc) on a plane, for 2D-look nodes
           in a 3D scene (gives "2.5D" look when camera tilted).

  [ ] 1.13 drawResourceNode(pos, size, filePath, meta?)   (M2-4)
           → Small 3D box with file icon + filename label.
           → Clicking opens a thumbnail or inline mesh viewer
             (loads the .obj via OBJLoader into a preview
             sub-scene or thumbnail sprite).
           → Color = domain color for the fact that references it.

  [ ] 1.14 drawToolEdge(from, to, toolName, width)  (M2-4)
           → Distinct visual from rule edges: orange color,
             dashed pattern, tool icon at midpoint.
           → Hover shows "called: mesh_analyzer.py".

  [ ] 1.15 drawEClassNode(pos, size, memberCount)  (M2-4)
           → Rounded rectangle / pill shape.
           → Shows count of equivalent members.
           → Hover expands to show member list.

  [ ] 1.16 drawPortalNode(pos, size, templateLabel)  (M2-4)
           → Diamond shape or highlighted ring.
           → Template text like "∀x. x+0=x".
           → Edges from portal to instantiations are thin dashed.

  [ ] 1.17 drawDomainRegion(center, extent, domainColor, label) (M2-4)
           → Semi-transparent colored plane or bounding box
             that groups nodes of the same domain.
           → Optional: toggle domain regions on/off.

PHASE 2 — Graph Layout Engine
------------------------------------------------------------
  [ ] 2.1  Circular layout (already stubbed)
           - nodes on circle perimeter
           - sorted by id / depth / cluster
           - radius scales with node count
           ✓ works for <~30 nodes

  [ ] 2.2  Fitch-style vertical proof layout
           - each proof line is a row, y decreasing downward
           - hypothesis lines: x offset proportional to depth
           - justified lines: x = base offset
           - vertical proof bars: Line2 or BoxGeometry
             spanning the scope vertically
           - supports nested subproofs (indentation depth)

  [ ] 2.3  Tree / DAG layout (ETPS TABLEAU style)
           - forward: root at top/bottom, branches downward/upward
           - backward: goal at top, subgoals branch downward
           - horizontal spacing to avoid collisions
           - optional orthogonal (routing with right angles)
           - optional curved edges

  [ ] 2.4  Force-directed layout (d3-force-inspired)
           - repulsion + attraction + gravity
           - iterate until convergence or max steps
           - pin nodes by depth to enforce layer ordering
           - suitable for large proof dependency graphs

  [ ] 2.5  Dependency graph layout (Lean/Mathlib style)
           - topological sort for node layers
           - hierarchical: theorem depends on lemmas below
           - Sugiyama-style layered layout option

  [ ] 2.6  Layout manager / auto-detect
           - "circular" for small flat graphs
           - "fitch" for linear proofs (has HYP/PLAN/JUSTIFIED)
           - "tree" for tableau / backward proofs
           - "force" for large unstructured graphs
           - user can switch via UI or keyboard

  [ ] 2.7  Animated transitions between layouts
           - lerp node positions over ~500ms
           - optionally animate edges

PHASE 5 — Proof Visualization Modes
------------------------------------------------------------
  These correspond to the different proof styles in ETPS
  and Lean/Mathlib.

  [ ] 5.1  Fitch-style proof view (ETPS default)
           - Vertical list of proof lines
           - Horizontal axiom/hypothesis bars
           - Vertical scope lines for subproofs
           - Line numbers on the left
           - Rendered as 3D geometry (boxes + lines)
             in a tilted camera for "2.5D" look

  [ ] 5.2  Tableau / tree view (ETPS TABLEAU)
           - Root at top, branches downward
           - Each branch is a subgoal decomposition
           - Closed branches (marked ⊥) colored red
           - Open branches (not yet closed) colored yellow
           - Completed branches colored green

  [ ] 5.3  Tactic proof view (Lean-style InfoView)
           - Goal state before tactic
           - Tactic name as edge label
           - Goal state after tactic
           - Subgoals fan out as children
           - Supports sequential (linear tactic blocks)
             and structured (case/induction with named goals)

  [ ] 5.4  Dependency graph view (Mathlib-style)
           - Nodes: theorems/lemmas/definitions
           - Edges: "depends on" (bidirectional for usage)
           - Layout: topological layers
           - Zoom out to see large-scale dependency web
           - Zoom in to see individual proof terms

  [ ] 5.5  Hybrid view mode
           - Main area: one of the above
           - Corner inset: zoomed-out minimap of full graph
           - Click minimap to pan

  [ ] 5.6  Resource mesh preview  (M2-4)
           - When a RESOURCE node references a .obj file:
           - Load the mesh via OBJLoader (Three.js addon)
           - Show a thumbnail in a small inset viewport
           - Label with metadata (units, mass, material)
             from the companion .meta JSON
           - Wireframe overlay option
           - Click to expand to full-size model viewer

  [ ] 5.7  Domain-layered view  (M2-4)
           - Nodes colored by domain
           - Each domain in its own z-layer (offset along
             the camera look-at axis, or color-separated)
           - Domain toggle buttons float at the edge of
             each layer
           - Edges that cross domains are highlighted
             (e.g. physics used a math result)

Frontend ↔ Backend Integration Note
------------------------------------------------------------
  The backend outputs a single JSON that contains:
  1. The full knowledge graph (nodes + edges with domains)
  2. An audit trail (compact winning path)
  3. Metadata (domains_used, tools_called, stats)

  The frontend renders the graph by default. The audit
  trail is used for the proof-stepper playback view.
  Selecting a node queries the graph for its detail.

  Domain-awareness:
  - Node colors come from the "domain" field
  - The sidebar domain switches filter by domain
  - Cross-domain edges are visually distinct
  - Resource nodes show file info + optional mesh preview

  JSON schema is defined in BACKEND_PLAN Phase 14.
