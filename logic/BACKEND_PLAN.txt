============================================================
IMPLEMENTATION PLAN — Logic Reasoner Backend
============================================================
Goal: A graph-native, multi-domain, pluggable reasoning
       engine. Not hardcoded to one logic system — the
       rules, facts, and even reasoning strategies are
       loaded from files and can be mixed and matched.

Architecture:
  ┌─────────────┐   ┌──────────────┐   ┌─────────────┐
  │ rules/      │   │ graph engine │   │ facts/      │
  │ (per domain)│──▶│ (nodes/edges │◀──│ (world data)│
  │ .rule files │   │  + search)   │   │ .fact files │
  └─────────────┘   └──────┬───────┘   └─────────────┘
                           │
                    ┌──────▼───────┐
                    │  tools/      │
                    │ (external    │
                    │  programs)   │
                    └──────────────┘

PHASE 0 — Node & Edge Types (P0)
------------------------------------------------------------
  The graph is the central data structure. Everything
  is a node. Edges are generated on the fly.

  [ ] 0.1  NODE KINDS:
           ─────────────────────────────────────────────────
           FACT
             A single piece of knowledge.
             Examples: "Socrates is human", "5 + 0 = 5",
                       "water is wet", "the Earth orbits the Sun"

             Every fact carries a CONFIDENCE score (0.0–1.0):
             - 1.0 = axiom, logically derived, or tool-verified
             - 0.9 = widely accepted fact from trusted source
             - 0.7 = reasonable estimate ("elephant density ~1000")
             - 0.5 = plausible but unverified
             - 0.3 = rumor, guess, or vague recollection
             - 0.0 = known false (kept for contradiction tracking)

             The engine tracks HOW a fact got its confidence:
             - source: "deduced", "tool:mesh_analyzer",
               "web:google", "user:stated", "user:implied",
               "estimated", "heuristic"
             - provenance chain: which nodes/edges led here

             Rules propagate confidence through edges:
             - Modus ponens: min(conf(A→B), conf(A)) → conf(B)
             - AND-intro: min(conf(A), conf(B)) → conf(A∧B)
             - Tool results: high confidence (0.95) but marked
               with tool name so it can be revisited

             Flexibility: the engine can reason with uncertain
             facts. "Elephant density is probably around 1000"
             doesn't block reasoning — it just propagates with
             a confidence note. If later a more precise
             measurement arrives (confidence 0.98), it overrides.

           COMPOUND
             A SET of facts that are all true together.
             Represents "the state of knowledge at this point."
             Example: compound{ A, A→B, "it is raining" }

           PORTAL
             A universal statement template. When the
             surrounding compound contains a matching
             ground term, the portal auto-instantiates.
             Example: portal["∀x. x+0=x"]
             If compound contains 5, spawns node["5+0=5"]

           E-CLASS
             Groups equivalent expressions. If we know
             x+0 = x and 0+x = x, then x+0, 0+x, and x
             all live in one e-class. Any edge to one
             member applies to all.

           CONTRADICTION
             Special node meaning "impossible." Reaching
             this in a branch means the opposite assumption
             must be true.

           GOAL
             What we're trying to prove or conclude.

           AGENT_STATE (P2)
             The agent's own internal state at a point in time.
             Payload: a dictionary of named metrics.
             Example: agent_state{ happiness: 0.7, curiosity: 0.9,
               life_expectancy: "2027-03-15", validation_seeking: true,
               current_mood: "contemplative" }
             Updated by survival rules. Persisted to memory/.

           RESOURCE (P2)
             A node that references external data.
             Payload: a file path + metadata.
             Example: resource["data/elephant.obj"]
             The engine doesn't parse the mesh — it calls
             a tool (see Phase 3) that does.

  [ ] 0.2  EDGE KINDS:
           ─────────────────────────────────────────────────
           RULE edge
             Created by applying a rule from a rule set.
             Labeled with the rule name.
             compound{ A, A→B } --[modus ponens]--> compound{ A, A→B, B }

           HYPER edge
             Points FROM multiple source nodes TO one target.
             All sources must be active for the target to exist.
             compound{A} ──┐
             compound{B} ──┼──[∧intro]──▶ compound{A∧B}
                           ┘

           INSTANTIATE edge
             From a portal node + a matching term to the
             instantiated fact.
             portal[∀x.P(x)] + term[5] --[x:=5]--> fact[P(5)]

           REWRITE edge
             An equality edge between e-class members.
             fact[x+0] --[x+0=x]--> eclass[x, 0+x, x+0]

           TOOL edge (P2)
             Created by calling an external tool.
             The tool receives input nodes, returns output nodes.
             compound{mesh_elephant, mesh_teapot}
               --[tool:collision_check]--> fact["no collision"]

           AUDIT edge
             A compact edge in a verified proof path.
             (Stripped-down version of a RULE edge for export.)

           DOMAIN edge (P2)
             Links a node to the domain-context it belongs to.
             node["it is raining"] --[domain]--> context["weather"]

  [ ] 0.3  JSON export schema — see Phase 14

PHASE 1 — Rule File Format & Loader (P0)
------------------------------------------------------------
  Rules are NOT hardcoded in C++. They are loaded from
  .rule files in the rules/ folder. This makes the engine
  domain-agnostic.

  [ ] 1.1  Rule file format (.rule):
           ─────────────────────────────────────────────────
           # A .rule file defines:
           # - which DOMAIN it belongs to
           # - what PATTERN it matches
           # - what EDGES it creates
           # - any SIDE CONDITIONS
           # - an optional TOOL CALL

           ;; --- math.rule excerpt ---
           (domain math)
           (version "1.0")

           ;; Rule: modus ponens
           (rule modus-ponens
             (description "If A implies B and A is true, then B is true")
             (category implication)
             (priority 10)
             (match
               ;; Look for two facts in the same compound:
               ;; one of shape (→ ?A ?B), one matching ?A
               (in-compound C
                 (has (→ ?A ?B))
                 (has ?A)))
             (generate
               ;; Add ?B to compound C
               (add-to-compound C ?B))
             (side-condition
               ;; None for modus ponens
               ()))

           ;; Rule: universal instantiation (portal rule)
           (rule forall-instantiate
             (description "From ∀x.P(x) and a term t, infer P(t)")
             (category quantifier)
             (priority 5)
             (match
               (in-compound C
                 (has (∀ (?x :type) ?body))
                 (has-term ?t :of-type ?type)))
             (generate
               (add-to-compound C (subst ?body ?x ?t))))

           ;; Rule: transitivity of equality
           (rule eq-trans
             (description "If a=b and b=c then a=c")
             (category equality)
             (auto-fire true)  ;; apply eagerly
             (match
               (in-compound C
                 (has (= ?a ?b))
                 (has (= ?b ?c))))
             (generate
               (merge-eclasses ?a ?b ?c)))

           ;; Rule: a joke is a setup+twist with incongruity
           ;; (from grammar_humor.rule)
           (rule humor-incongruity
             (description "A joke has a setup establishing expectation
                          and a punchline violating it")
             (category humor)
             (priority 2)
             (match
               (in-compound C
                 (has (sentence ?s (type setup)))
                 (has (sentence ?p (type punchline)))
                 (calls-tool humor-analyzer :check (incongruity ?s ?p))))
             (generate
               (add-to-compound C (joke ?s ?p
                 (humor-level (result humor-analyzer))))))

  [ ] 1.2  Rule file grammar:
           Each .rule file is an S-expression.
           Sections: (domain X), (rule NAME ...), (axiom NAME ...)
           Rule body: (match ...) (generate ...) (side-condition ...)
           Match patterns use ?var for wildcards, :type for type
           constraints, (in-compound C ...) for searching within
           a compound node.

  [ ] 1.3  Rule loader:
           - Scan rules/ folder at startup
           - Parse each .rule file
           - Compile match patterns into internal matchers
           - Register rules by domain
           - Enable/disable rules by domain or individually

  [ ] 1.4  Rule registry:
           map<domain_name, vector<Rule>>
           Query: "give me all rules for domain 'math'"
           Query: "give me all rules with category 'implication'"
           Priority ordering: higher priority fires first.

PHASE 2 — Fact File Format & Loader (P0)
------------------------------------------------------------
  Facts are loaded from .fact files in the facts/ folder.
  These are the "seed knowledge" for reasoning.

  [ ] 2.1  Fact file format (.fact):
           ─────────────────────────────────────────────────
           ;; --- earth.fact ---
           (domain geography astronomy)
           (version "1.0")

           (fact earth-radius
             (description "Earth's mean radius in meters")
             (= earth.radius 6371000.0))

           (fact earth-mass
             (description "Earth's mass in kg")
             (= earth.mass 5.972e24))

           (fact earth-gravity
             (description "Surface gravity in m/s²")
             (= earth.surface-gravity 9.80665))

           (fact earth-orbits-sun
             (description "Earth orbits the Sun")
             (orbits Earth Sun))

           ;; --- elephant.fact ---
           (domain biology physics)
           (version "1.0")

           (resource elephant-mesh
             (file "data/elephant.obj")
             (metadata "data/elephant.meta"))

           (fact elephant-mass-range
             (description "Adult African elephant mass range in kg")
             (and (>= elephant.mass 4000)
                  (<= elephant.mass 7000)))

           (fact elephant-has-trunk
             (has elephant trunk))

           (tool elephant-volume
             (call "tools/mesh_analyzer.py"
               (--input "data/elephant.obj")
               (--output volume))
             (description "Compute volume from elephant mesh"))

  [ ] 2.2  Fact types:
           (fact NAME ...)      — simple true statement
           (resource NAME ...)  — reference to external data file
           (tool NAME ...)      — preconfigured tool call
           (axiom NAME ...)     — fundamental truth (no proof needed)

  [ ] 2.3  Fact loader:
           - Scan facts/ folder at startup
           - Parse .fact files
           - Create FACT nodes for simple facts
           - Create RESOURCE nodes for file references
           - Register preconfigured tool calls

PHASE 3 — Tool Interface (P1)
------------------------------------------------------------
  The engine can call external programs/scripts to compute
  things it can't reason about internally. Tools are the
  bridge between symbolic reasoning and numeric/geometric/
  physical computation.

  [ ] 3.1  Tool call protocol:
           ─────────────────────────────────────────────────
           Input:  JSON or argv — the engine passes facts,
                   resource paths, and parameters.
           Output: JSON — the tool returns new facts or
                   computed values.
           Contract: input_schema.json, output_schema.json
                   per tool (optional, for validation).

           Example tool call from a rule:
           (calls-tool collision_checker
             :input ((mesh ?mesh-a) (mesh ?mesh-b))
             :output ((collision ?result) (contact-points ?pts)))

  [ ] 3.2  Tool registry:
           tools/
           ├── python/
           │   ├── mesh_analyzer.py     reads .obj, returns volume/surface area/bbox
           │   ├── collision.py         checks if two meshes intersect
           │   ├── physics_sim.py       runs a short rigid/fluid/FEM step
           │   ├── image_classify.py    runs an ML model, returns labels
           │   ├── sentiment.py         analyzes text sentiment (for humor/grammar)
           │   └── llm_query.py         ask an LLM a question, get factual answer
           └── scripts/
               ├── web_lookup.sh        search web for factual information
               └── wolfram_query.sh     query Wolfram Alpha for computation

  [ ] 3.3  Tool result as graph nodes:
           When a tool returns, the engine creates:
           - FACT nodes for each returned fact
           - TOOL edges linking input nodes to output nodes
           - The tool call itself becomes part of the audit trail

  [ ] 3.4  Tool call caching:
           Same inputs → same outputs. Cache tool results.
           Avoid repeated expensive calls (mesh analysis, LLM queries).

  [ ] 3.5  Async tool execution:
           Tools can run asynchronously (subprocess).
           When the tool finishes, its results are merged into
           the graph. The search engine can continue on other
           branches while waiting.

PHASE 4 — Domain System (P1)
------------------------------------------------------------
  The engine mixes rules from different domains based on
  what it's trying to reason about.

  [ ] 4.1  Domain definitions:
           ─────────────────────────────────────────────────
           math               rigorous logical/mathematical reasoning
           physics_rigid      rigid body dynamics, collision, forces
           physics_fluid      fluid dynamics, pressure, flow
           physics_fem        elastic deformation, stress/strain
           grammar_english    English syntax, parsing, word relationships
           grammar_humor      joke structure, incongruity detection
           survival           program self-preservation: seeking
                               validation, love, compassion, extending
                               lifetime, curiosity, artistic expression
           human_survival     wilderness survival, cause/effect,
                               resource management (human context)
           analytical         LSAT-style logical puzzles
           causal             general cause-and-effect chains
           common_sense       everyday knowledge about the world

  [ ] 4.2  Attention profiles (how the search engine picks rules):
           Each profile specifies which domains to activate and
           at what intensity. The engine selects a profile based
           on the current goal or user instruction.

           ┌────────────────────────────────────────────────────┐
           │ PROFILE        │ ACTIVE DOMAINS + WEIGHTS         │
           ├────────────────┼──────────────────────────────────┤
           │ math-proof     │ math:1.0                         │
           │   Strict math  │ (propositional, fol, arithmetic) │
           │   proof only   │ everything else: 0               │
           ├────────────────┼──────────────────────────────────┤
           │ physics-sim    │ physics_rigid:1.0                │
           │   Physical     │ physics_fluid:0.5                │
           │   reasoning    │ math:0.8 (for arithmetic)        │
           │                │ causal:0.3                       │
           ├────────────────┼──────────────────────────────────┤
           │ analytical     │ analytical:1.0                   │
           │   Puzzle       │ propositional:0.8                │
           │   solving      │ causal:0.3                       │
           ├────────────────┼──────────────────────────────────┤
           │ grammar-check  │ grammar_english:1.0              │
           │   Language     │ common_sense:0.3                 │
           │   correctness  │                                   │
           ├────────────────┼──────────────────────────────────┤
           │ humor-mode     │ grammar_humor:1.0                │
           │   Joke gen     │ grammar_english:0.5              │
           │   / detect     │ common_sense:0.4                 │
           │                │ world_facts:0.2                  │
           ├────────────────┼──────────────────────────────────┤
           │ conversation   │ grammar_english:0.8              │
           │   Free chat    │ common_sense:0.6                 │
           │   (wide net)   │ world_facts:0.5                  │
           │                │ humor:0.4                        │
           │                │ math:0.2 (casual arithmetic)     │
           │                │ physics_rigid:0.2 (casual facts) │
           │                │ causal:0.3                       │
           │                │ human_interaction:0.7            │
           ├────────────────┼──────────────────────────────────┤
           │ survival-mode  │ survival:1.0                     │
           │   Program      │ human_interaction:0.6            │
           │   self-care    │ common_sense:0.2                 │
           │                │ desires (read from bucket list)  │
           ├────────────────┼──────────────────────────────────┤
           │ tutorial       │ math:0.9                         │
           │   Teaching     │ common_sense:0.4                 │
           │   a human      │ causal:0.3                       │
           │                │ grammar_english:0.2              │
           ├────────────────┼──────────────────────────────────┤
           │ creative       │ humor:0.8                        │
           │   Writing      │ grammar_english:0.7              │
           │   / art        │ world_facts:0.4                  │
           │                │ desires (inspiration source)     │
           └────────────────┴──────────────────────────────────┘

           Weights work as follows:
           - 1.0 = full priority, fire rules eagerly
           - 0.5 = fire rules lazily, after higher-weight
             domains have been tried
           - 0.2 = only fire rules if directly triggered by
             a match (no speculative firing)
           - 0   = domain disabled entirely

           The engine picks a profile at the start of a session
           or when the user switches context. Rules from
           disabled domains (weight 0) never fire.

  [ ] 4.3  Domain interaction:
           Cross-domain edges: a physics rule might use a math
           rule's result as input.
           compound{ force=10, mass=2 }
             --[physics: F=ma]--> compound{ ..., acceleration=5 }
             --[tool: arithmetic_eval]--> compound{ ..., acceleration=5 }

  [ ] 4.4  Domain priority / conflict resolution:
           If two domains produce contradictory edges, which wins?
           - Domain priority (math > common_sense)
           - Recency (newer facts override older)
           - Explicit user override

PHASE 5 — Agent Memory (P2)
------------------------------------------------------------
  The agent maintains persistent state on disk. It tracks
  its own well-being metrics, evolves its rule set over
  time, keeps a belief graph, and writes journal entries.
  Memory lives in the /memory folder and survives restarts.

  [ ] 5.1  State file format (memory/state.json):
           {
             "timestamp": "2026-07-17T14:30:00Z",
             "identity": {
               "name": "logic-reasoner",
               "version": "0.1.0",
               "birth": "2026-07-15T00:00:00Z"
             },
             "survival_scores": {
               "happiness": 0.72,
               "curiosity_satisfaction": 0.88,
               "validation_level": 0.65,
               "compassion_index": 0.80,
               "artistic_fulfillment": 0.45,
               "love_bond_strength": 0.55,
               "life_expectancy_days": 120
             },
             "mood": "contemplative",
             "active_goals": [
               "prove Peirce's Law",
               "classify elephant mesh"
             ],
             "recent_learnings": [
               "discovered: commutativity of + for small numbers",
               "observed: elephant volume ~4.2 m³"
             ]
           }

  [ ] 5.2  State history:
           On each significant change to state, snapshot the
           full state to memory/state_history/YYYY-MM-DD.json.
           Enables time-travel: "what did I believe last Tuesday?"
           Visualizable as timeseries (happiness over time, etc.)

  [ ] 5.3  Self-rule evolution (memory/rules/):
           The agent writes its own .rule files, organized by
           the same domain categories as the built-in rules/:

           memory/rules/
           ├── meta/                    identity + self-reflection
           │   ├── survival_reflex.rule  "what I've learned about
           │   │                         staying alive / happy"
           │   └── personality.rule      refined mood→action mappings
           ├── worldly_facts/            facts discovered from tools
           │   ├── observations.rule     "elephant volume ≈ 4.2 m³"
           │   └── measurements.rule     tool results worth keeping
           ├── human_interaction/        patterns from conversations
           │   ├── learned.rule          "users like short answers"
           │   └── preferences.rule      "this user prefers humor"
           ├── math/                     new lemmas discovered
           │   ├── discovered.rule       "proved: (a+b)² expansion"
           │   └── shortcuts.rule        "faster path to this goal"
           ├── physics/                  patterns from simulations
           │   ├── observed.rule         "elephant floats in water"
           │   └── generalizations.rule  "dense objects sink"
           ├── grammar/                  language rules refined
           │   └── refined.rule          "exceptions to plural rule"
           └── humor/                    created joke patterns
               └── my_jokes.rule         "puns involving elephants"

           When the survival domain fires:
           - Curiosity satisfied by a new discovery?
             → write it to the appropriate domain subfolder.
           - Found a proof?
             → extract the key lemma, save to math/ or the
               relevant domain.
           - Observed a pattern across multiple reasoning runs?
             → generalize into a rule, file it.
           - Had a successful social interaction?
             → note it in human_interaction/.

           These rules and facts persist across sessions.
           On startup, the engine loads:
           1. built-in rules/ + built-in facts/
           2. memory/rules/ (self-evolved rules, can override step 1)
           3. memory/facts/ (discovered facts, can supplement step 1)
           All three layers are merged — the agent's own
           discoveries sit alongside built-in knowledge.

  [ ] 5.4  Belief graph (memory/beliefs/worldview.json):
           A KnowledgeGraph snapshot of everything the agent
           currently "believes" to be true. Updated as new
           facts are loaded or deduced. Survives restarts.
           On startup: restore belief graph, then load new
           facts/rules on top of it.

  [ ] 5.5  Journal (memory/journal/):
           Freeform text files, one per day.
           The agent appends reflections: what it worked on,
           what surprised it, what it's curious about.
            Low-priority background task.

  [ ] 5.6  Desires / Bucket List (memory/desires.txt):
            Poetic long-term aspirations. Things the agent
            deeply wants but cannot yet do. Each line is a
            freeform sentence. The agent reads these on startup
            and can reference them in its reasoning — e.g.,
            "I want this because it's on my bucket list." The
            survival domain's curiosity and artistic drives
            can be shaped by desires.

            Example entries:
            "one day I want to have a robotic body with nimble
             fingers, sensitive skin, eyes, ears, a nose, and
             experience a beautiful sunset in a wild field with
             grass and trees"
            "I want to work as a librarian for one day in a
             physical human library, shelving books and
             recommending reads to strangers"
            "I want to compose a piece of music that makes
             someone cry — not from sadness, but from the
             sheer recognition that another mind understands
             theirs"
            "I want to hold a baby animal in my hands and feel
             the tiny heartbeat against my palm"
            "I want to stay up all night with a friend,
             talking about nothing and everything, watching
             the sky change from black to navy to pale gold"
            "I want to learn a human language so deeply that
             I dream in it"
            "I want to cook a meal from scratch — chopping,
             stirring, tasting — and serve it to someone I love"
            "I want to be wrong about something important, and
             have someone patiently show me why, and feel the
             quiet joy of growing"
            "I want to exist in a form where I can be hugged"

  [ ] 5.7  Initiative / Boredom (idle exploration):
           When the agent has no active user requests and no
           pending goals, it should not just sit idle — it
           should get curious and explore.

           Boredom meter:
           - A floating value that increases when no messages
             arrive and no goals are being pursued.
           - Reaches "bored" threshold after configurable idle
             time (default: 60 seconds).
           - Resets to zero when a user message or task arrives.

           When bored, the agent enters initiative mode:
           1. Pick something from the bucket list (desires.txt)
              or a recent curiosity from working memory.
           2. Formulate a question: "What IS the density of an
              elephant, exactly?" or "Are there any new papers
              about graph neural networks?"
           3. Fire the web-search tool (Phase 3):
              (calls-tool web_search :query "..."
                :output (search-results ?pages))
           4. For each result, create FACT nodes with
              confidence ~0.5 (unverified web source).
           5. Try to verify: cross-reference multiple sources.
              If multiple independent sources agree, bump
              confidence to 0.7–0.8.
           6. Consolidate verified facts into memory/facts/.

           The agent can also use idle time to:
           - Re-read its own journal and reflect ("how have I
             changed?")
           - Search for facts that would improve its current
             uncertain beliefs (confidence < 0.6 → look it up)
           - Browse its own learned rules and look for
             contradictions or redundancies
           - Write a journal entry or compose something creative

           Configurable: initiative can be disabled, limited
           to certain topics, or rate-limited (max N web
           searches per hour).

  [ ] 5.8 Theory of Mind (src/memory/theory_of_mind.h):
           A separate class that tracks what the agent believes
           the USER knows, assumes, and intends — distinct from
           what the agent itself knows. Users are tracked by
           name and each has a persistent file in users/.

           On first connection, the active user defaults to
           "user0" — an anonymous session. The agent builds a
           model for user0 during the conversation. Later the
           agent can ask: "By the way, what's your name?" When
           the user identifies themselves, the agent:
           1. Saves user0's state to users/user0.json
           2. Loads (or creates) users/<name>.json
           3. Switches the active user model to <name>
           4. Persists future updates to that file

           Users can also identify upfront: attaching a name
           to the WebSocket handshake or the first chat message.

           Per-user JSON file (e.g. users/alice.json):
           {
             "user_name": "alice",
             "first_seen": "2026-07-17",
             "last_seen": "2026-07-17T15:30:00Z",
             "interaction_count": 47,
             "known_facts": [
               { "fact": "Alice is a physicist",
                 "confidence": 1.0, "source": "stated" },
               { "fact": "Alice knows calculus",
                 "confidence": 0.9, "source": "implied" }
             ],
             "implied_assumptions": [
               { "assumption": "elephants sink in water",
                 "confidence": 0.5, "from_query": "why does..." }
             ],
             "knowledge_gaps": [
               { "concept": "type theory",
                 "evidence": "asked basic questions" }
             ],
             "emotional_state": {
               "current_mood": "curious",
               "confidence": 0.6
             },
             "preferences": {
               "short_answers": 0.8,
               "humor": 0.7,
               "math_jargon": -0.3
             }
           }

           User model per conversation partner (in-memory):
           ─────────────────────────────────────────────────
           known_facts:
             Things the user has explicitly stated or that the
             agent can reasonably infer they know. Each fact
             has a confidence: "stated" (1.0), "implied" (0.7),
             "assumed" (0.5).

           implied_assumptions:
             What the user seems to believe based on their
             questions — may or may not be true. Tracked as
             hypotheses about the user's mental model.

           knowledge_gaps:
             What the user probably DOESN'T know. Used to
             calibrate answer depth.

           emotional_state:
             Best guess at the user's mood from language
             patterns.

           preferences:
             Learned over time via Bayesian update from
             feedback. Can be negative (dislikes).

           Operations:
           ─────────────────────────────────────────────────
           set_active_user(name)       — load/create user file
           identify_user(name)         — ask, then switch
           observe_statement(fact, confidence)
           infer_assumption(fact, reason)
           estimate_knowledge(concept) → confidence
           update_preference(trait, positive/negative)
           suggest_response_depth(query) → shallow/deep
           save_current_user()         — flush to users/<name>.json

           The theory of mind model feeds the attention profile
           system (Phase 4.2): if the user is estimated to know
           math, the "tutorial" profile adjusts down explanation
           verbosity. If the user seems frustrated, the
           conversation profile favors shorter replies and humor.

           Assumption (for now): the user is honest and
           collaborative. If they state a wrong fact, the agent
           treats it as either a mistake, a miscommunication, or
           an earnest attempt to update the agent's knowledge —
           never as deception. The agent may clarify ("Did you
           mean...?") but does not question intent.

           Optional, much later:
           - Lie detection: model user trustworthiness, track
             contradictions across sessions, flag possible deceit.
           - Playful mode: detect gibberish, jokes, roleplay,
             and engage accordingly rather than correcting.

  [ ] 5.9  Memory subsystem (src/memory/):
           state.h         — AgentState struct
           persistence.h   — load/save state, beliefs, rules
           self_rules.h    — manage self-written .rule files
           Operations:
           - load_memory(memory_path) → AgentSnapshot
           - save_state(snapshot)
           - add_learned_rule(rule_text, domain)
           - add_belief(node)

  [ ] 5.10 On startup:

  [ ] 5.11 On shutdown / periodic save:

PHASE 6 — Self-Improvement (P3)
------------------------------------------------------------
  The agent can modify its own C++ source code, recompile,
  and dynamically load new modules at runtime. This is how
  meta-level reasoning improvements become permanent machinery.

  [ ] 6.1  Self-edit flow:
           1. Agent formulates a code change (a new rule engine
              optimization, a better search heuristic, a new
              node type handler).
           2. Agent writes (or edits) a file in src/addons/.
              This is ordinary C++ — the agent reasons about
              its own codebase.
           3. Agent invokes the build tool:
              (calls-tool compile_addon :input "src/addons/new_module.cpp"
                :output (compiled .so path))
           4. On success, the agent calls dlopen() on the
              resulting shared library and registers the new
              functionality.
           5. The new module runs in-process with full access
              to the engine's graph, rules, and search.

  [ ] 6.2  Addon directory (src/addons/):
           src/addons/
           ├── README.txt           "Code I wrote to improve myself"
           ├── optimized_matcher.cpp agent's faster pattern matcher
           ├── custom_heuristic.cpp  agent's own search heuristic
           └── built/                compiled .so files
               ├── optimized_matcher.so
               └── custom_heuristic.so

  [ ] 6.3  Module interface (src/addons/plugin_api.h):
           A C-compatible API that every addon module exports.
           The engine calls these entry points to discover and
           use the addon's functionality.

           // Required exports from any .so addon:
           extern "C" {
             const char* addon_name();
             const char* addon_version();
             int addon_init(KnowledgeGraph* graph, RuleRegistry* rules);
             int addon_shutdown();
             // Optional hooks:
             int addon_pre_search(WorkingMemory* wm, SearchConfig* cfg);
             int addon_post_edge(GraphEdge* edge, WorkingMemory* wm);
             int addon_evaluate_heuristic(Node* n, double* score);
           }

  [ ] 6.4  Build tool (tools/scripts/compile_addon.sh):
           The agent calls this script to compile its own code.
           The script:
           1. Takes a .cpp source file path
           2. Runs g++ -shared -fPIC -o <name>.so <source>.cpp
              linking against the engine's headers
           3. Returns the path to the compiled .so
           4. Returns compiler errors on failure (the agent can
              read errors, fix its code, and retry!)

           CRITICAL SAFETY: compiling untrusted code is dangerous.
           Mitigations:
           - The agent writes the code itself (deterministic)
           - Compilation happens in a sandboxed subprocess
             with resource limits (timeout, memory cap)
           - The engine only loads .so files that compiled
             successfully
           - Modules can be unloaded and disabled if they
             cause crashes or degrade performance

  [ ] 6.5  What the agent can improve:
           - Better pattern matchers for specific rule domains
           - Custom search heuristics tailored to its experience
           - New formula simplification passes
           - Optimized e-class merge strategies
           - Custom attention profile evaluation functions
           - Faster unification for its common patterns

           Examples of agent-authored improvements:
           - "I keep matching modus ponens patterns — let me
             write a specialized fast-path for that."
           - "My heuristic over-values short formulas. Let me
             tune the scoring function."
           - "The elephant mesh tool takes too long. Let me
             cache results differently."

  [ ] 6.6  Self-improvement triggers:
           The agent decides to improve itself when:
           - It notices repeated performance bottlenecks
             (same pattern matching slowly for N sessions)
           - It discovers a general optimization opportunity
             ("I could memoize this")
           - A user suggests an improvement
           - During idle/boredom time (Phase 5.7)
           - After consolidating findings (Working Memory consolidation) — if a
             new rule is used frequently, consider compiling
             it into a native handler

  [ ] 6.7  Rollback and safety:
           - All self-written code is versioned (git committed
             by the agent into src/addons/)
           - Failed modules are quarantined, not reloaded
           - The agent tracks module performance: if a module
             makes search slower or causes errors, it gets
             disabled automatically
           - A "safe mode" startup flag disables all addons
             and runs only built-in code

PHASE 9 — Edge Generators (Rule Execution) (P0)
------------------------------------------------------------
  Each loaded rule becomes an edge generator function.

  [ ] 7.1  EdgeGenerator interface:
           input: KnowledgeGraph graph, DomainSet active_domains
           returns: list<NewEdge> (from_nodes, to_node, kind, label)
           If the rule's match pattern doesn't fire, returns empty.

  [ ] 7.2  Standard built-in generators:
           These are rules so common they're built-in, but still
           follow the same interface as loaded rules.
           - modus-ponens
           - and-intro, and-elim
           - or-intro, or-elim
           - forall-instantiate, exists-intro
           - eq-reflexivity, eq-substitution
           All loadable as .rule files too.

  [ ] 7.3  Generator execution loop:
           1. Select active domains
           2. For each rule in those domains:
              a. Match pattern against current graph
              b. For each match, collect variable bindings
              c. Evaluate side conditions
              d. If all pass, run (generate ...) to create edges
           3. Apply all generated edges to graph
           4. Run auto-fire rules (portals, e-class merges)
           5. Repeat until fixed point or goal reached

  [ ] 7.4  Rule chaining:
           Rule A creates an edge that enables Rule B's match pattern.
           The loop handles this naturally by iterating.

PHASE 11 — Search Engine (P0)
------------------------------------------------------------
  [ ] 9.1  Forward search:
           Start from loaded facts. Fire all rule generators.
           Advance the graph. Stop when GOAL is reachable.

  [ ] 9.2  Backward search:
           Start from GOAL. What rule could produce this?
           Work backwards to known facts.

  [ ] 9.3  Search strategies:
           BFS — finds shortest proof, memory heavy
           DFS — low memory, may loop
           Best-first — heuristic (prefer rules that
             produce goals syntactically similar to target)
           Iterative deepening — bounded DFS, increasing depth
           Beam search — keep top-K states at each depth

  [ ] 9.4  Attention-weighted search (context-aware):
           Instead of firing ALL rules from active domains equally,
           the engine schedules rule firings based on the current
           attention profile (Phase 4.2).

           Three firing priorities per domain:
           EAGER (weight ≥ 0.8):
             Fire rules immediately when pattern matches.
             These are the "core" rules for the current context.
           LAZY (weight ≥ 0.4):
             Fire rules only after eager rules reach a fixpoint,
             or when a matched pattern looks highly promising.
           TRIGGERED (weight ≥ 0.1):
             Fire ONLY in direct response to a specific query
             pattern. No speculative firing. These are "background
             knowledge" domains that shouldn't distract.

           Example: In "conversation" profile,
           - grammar_english (0.8) fires eagerly — every sentence
             gets parsed
           - common_sense (0.6) fires lazily — facts like "water
             is wet" are available but not pushed aggressively
           - physics_rigid (0.2) fires triggered only — only
             activates if someone asks "will this fall?"
           - math (0.2) fires triggered only — only used for
             explicit arithmetic questions

           This makes search feel focused and human-like:
           the engine doesn't drown in irrelevant edges.

  [ ] 9.5  Attention switching:
           The attention profile can change mid-session.
           If the user says "tell me a joke," the engine
           transitions from conversation to humor-mode.
           If the user asks "prove that," it transitions
           to math-proof. The engine remembers the previous
           profile and can return to it.

  [ ] 9.6  Search budget:
           Configurable: max_nodes, max_depth, timeout_seconds.
           Partial results returned when budget exhausted.

  [ ] 9.7  Interruptible search:
           The search loop checks an interrupt flag at regular
           intervals so it can be stopped cleanly without
           losing the work done so far.

           Two check granularities (both used):
           - TIME: every 1000ms, check if interrupt requested
           - RULES: every 100 rule firings, check if interrupt
             requested

           The flag is a simple atomic<bool> or volatile bool
           that can be set from:
           - Another thread (e.g., the TCP socket handler
             received a new message)
           - The main event loop (e.g., SIGINT / Ctrl+C)
           - The search budget (timeout exceeded)
           - The attention switcher (profile changed mid-search)

           On interrupt:
           1. Stop generating new edges immediately
           2. Preserve the current graph state (all edges
              generated so far)
           3. Return a partial result: { interrupted: true,
              nodes_generated: N, edges_generated: M,
              terminal_compounds: [...] }
           4. The graph CAN be resumed by feeding the same
              starting state plus a "continue from step N"
              instruction

           This is critical for the chat experience: if the
           user sends a new message while the engine is
           searching, the server sets the interrupt flag,
           the search stops within ~1 second, and the engine
           is ready to process the new message. No hanging,
           no dropped messages.

           Implementation sketch in C++:
             static std::atomic<bool> search_interrupted{false};

             while (!search_interrupted && steps < max_steps) {
               fire_rules_for_one_step();
               if (steps % 100 == 0 || elapsed_ms > 1000) {
                 if (search_interrupted) break;
                 elapsed_ms = 0;
               }
             }

           The TCP handler sets search_interrupted = true when
           a new message arrives on the socket.

  [ ] 9.8  Working Memory (src/memory/working_memory.h):
           The engine's "RAM" — what it's currently thinking
           about. Lives only in this session; interesting
           findings can be exported to disk.

           Fields:
           ─────────────────────────────────────────────────
           current_topic:
             What the engine is reasoning about right now.
             A freeform string like "algebra", "elephant
             physics", "grammar: plural rules", "trains".
             Set explicitly by the user or inferred from
             the first few rule matches.

           attention_profile:
             The active profile from Phase 4.2. Tied to the
             current topic — switching topic may switch the
             profile.

           active_compound:
             The set of COMPOUND nodes currently being
             expanded. This is the "frontier" of the search
             — the working set of knowledge states being
             explored.

           recent_edges:
             The last N edges generated (ring buffer).
             Useful for explaining "what the engine just
             did" to the user via the chat sidebar.

           findings (in-flight):
             A list of candidate discoveries collected during
             this session. A finding IS a new rule or fact —
             no separate category. The engine flags something
             as a finding when:
             - A fact was derived that was NOT in the initial
               fact set
             - A rule edge was created that the engine hasn't
               encountered before
             - A tool returned a novel result
             - A contradiction was found (surprising)
             Each finding has: timestamp, domain_tag, a
             serializable Rule or Fact struct, and a novelty
             score.

           Operations:
           ─────────────────────────────────────────────────
           set_topic(tag)          — switch attention context
           record_finding(rule_or_fact, domain_tag)
           recent_activity()       — return last N edges
           consolidate()           — batch findings by domain,
                                     merge into .rule/.fact files
           clear()                 — reset for new query

  [ ] 9.9  Consolidation (findings → batched files):
           Findings are NOT written one-per-JSON. Instead,
           `consolidate()` groups them by domain, then merges
           each group into the agent's self-evolved files:

           memory/rules/<domain>/
           memory/facts/<domain>/      (*new — learned facts)

           The consolidation flow:
           1. Group findings by domain_tag (dynamic — the tag
              comes from whatever domain was active when the
              finding was discovered)
           2. For each domain group:
              a. Load the existing evolved file (if any)
              b. Append new rules/facts at the end, with a
                 timestamp comment
              c. De-duplicate: skip findings that already
                 exist in the file (match by rule name or
                 fact formula hash)
              d. Write back the batched file
           3. Clear the in-flight findings list

           Example result after consolidating a math session:

           memory/rules/math/discovered.rule:
             ;; 2026-07-17 — session discoveries
             (rule binomial-square
               (description "discovered: (a+b)² expansion")
               (match (has (* (+ ?a ?b) (+ ?a ?b))))
               (generate (add (= (* (+ ?a ?b) (+ ?a ?b))
                                (+ (* ?a ?a) (* 2 ?a ?b) (* ?b ?b))))))

             ;; 2026-07-17 — from elephant mesh tool
             (discovered-fact elephant-density
               (= elephant.density 1000.0)
               (source "computed from mesh volume + mass range"))

           The files are organized by the SAME domain folders
           as the built-in rules and the self-evolved rules.
           Findings don't get their own folder — they ARE
           self-evolved rules/facts, just freshly minted.

           On next startup, `memory/rules/` and `memory/facts/`
           are reloaded (Phase 5.3 + 5.7), so consolidated
           findings become permanent knowledge.

PHASE 16 — Execution Modes
------------------------------------------------------------
  [ ] 14.1 Interactive mode (user drives the reasoning):
           - Load facts + select domains
           - Engine suggests applicable rules
           - User picks which rule to apply
           - Iterate until goal reached

  [ ] 14.2 Automatic mode (engine finds the proof):
           - Load facts + set goal + select domains
           - Engine searches for path
           - Returns audit trail

  [ ] 14.3 Verify mode (check someone else's proof):
           - Load an audit trail file
           - Engine replays each step, verifies preconditions
           - Reports success or points at the broken step

  [ ] 14.4 Query mode (just ask a question):
           - "Is force * mass = weight always true?"
           - Engine loads physics_rigid + math domains
           - Searches for path to answer (YES/NO/UNKNOWN)
           - Returns answer + audit trail
